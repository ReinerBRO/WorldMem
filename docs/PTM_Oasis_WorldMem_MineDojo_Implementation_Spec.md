# Predictive-Test Memory on Oasis/WorldMem-family World Models

**Purpose:** This document is a concrete engineering specification for implementing and validating **Predictive-Test Memory (PTM)** on an Oasis-family interactive Minecraft world model.

**Intended executor:** a local CLI coding agent.

**Core idea:** memory should not merely compress/retrieve past frames. A bounded memory state should be trained to preserve information that improves future action-conditioned rollout, especially loop return, landmark revisit, and object/state persistence.

---

## 0. Final decision

### Use this open-source project

**Primary repo to fork/modify:**

```bash
git clone https://github.com/xizaoqu/WorldMem.git
```

Why this repo:

1. It is explicitly an **Open-Oasis-family** memory world-model project.
2. It already contains training/evaluation entry points such as `main.py`, `train_stage_*.sh`, `evaluate.sh`, `infer.sh`, `data_generator.py`, `algorithms/`, `datasets/`, and `configurations/`.
3. Its README states that the architecture is based on Open-Oasis and uses pretrained VAE and DiT weights from Open-Oasis.
4. It is a direct nearby memory baseline: memory bank + memory attention for long-term consistency.
5. It is much more suitable than starting from `etched-ai/open-oasis` alone, because Open-Oasis is primarily inference code for Oasis 500M, not a complete memory training/evaluation project.

### Use this open-source model

**Base model / initialization:**

```text
Open-Oasis / Etched Oasis 500M weights
```

Use WorldMem's loading path where possible. WorldMem's README says to download pretrained weights from Oasis and shows inference loading either separate diffusion/VAE checkpoints or a combined checkpoint.

Practical priority:

1. First try WorldMem's expected checkpoint format:
   - diffusion model checkpoint
   - VAE checkpoint
   - combined checkpoint if available
2. If those do not load cleanly, use the Open-Oasis weights and adapt the loader minimally.
3. Do **not** train an Oasis-style world model from scratch.

### Use this data

**Do not use an uncertain public Minecraft trajectory dataset as the main training data.**

Primary PTM training data should be generated by **MineDojo**:

```text
MineDojo simulator rollout
→ RGB frames
→ actions
→ pose/location/yaw/pitch
→ inventory / inventory_change
→ voxels / local block state
→ programmatic event logs
→ programmatic future-test labels
```

This is not optional. PTM needs action-conditioned trajectory labels. If a public dataset lacks action, pose, event, and future-test labels, it is not the main training data.

## 1. Background and motivation

Existing memory world models mostly ask:

```text
How do we store the past?
How do we compress the past?
How do we retrieve relevant history?
```

Examples:

- **WorldMem** stores memory frames and states such as poses and timestamps, then uses memory attention to retrieve relevant memory for long-term consistency.
- **WorldPack** proposes compressed memory and retrieval for spatial consistency in video world modeling.
- **WorldKV** operates on KV cache retrieval/compression and is more model-agnostic, but it is an inference-time/training-free memory efficiency method.

PTM asks a different question:

```text
What should a world-model memory be trained to preserve?
```

Answer:

```text
A good memory is a bounded predictive state that improves future executable action-conditioned tests.
```

In this project, this must not remain a slogan. It must produce benchmark gains:

```text
long-horizon rollout quality improves
loop revisit consistency improves
landmark persistence improves
placed-object persistence improves
memory budget vs performance curve improves
```

## 2. Precise method definition

At time `t`, the model has history:

```text
h_t = (o_0, a_0, o_1, a_1, ..., o_t)
```

A memory encoder produces a bounded memory:

```text
m_t = M_theta(h_t)
```

A future executable test is:

```text
tau = (a_t, a_{t+1}, ..., a_{t+K-1}, q)
```

where `q` describes what outcome to test after executing the action suffix.

Examples:

```text
q = future frame embedding at t+K
q = whether the agent returns to a previously seen place
q = whether a landmark should be visible at return
q = whether a placed object should still exist
q = which previous frame is matched by the return view
```

A test decoder predicts:

```text
y_hat = D_phi(m_t, tau)
```

Training objective:

```text
L = L_world_model
  + lambda_test * L_future_tests
  + beta_bottleneck * L_bottleneck
```

Important implementation constraint:

```text
The future-test decoder must read m_t, not raw full history.
```

Otherwise, the test objective may bypass memory and fail to shape the memory state.

## 5. PTM model design

### 5.1 Memory encoder

Implement:

```python
class PredictiveTestMemory(nn.Module):
    def __init__(
        self,
        frame_dim,
        action_dim,
        memory_dim=1024,
        num_memory_tokens=16,
        num_layers=4,
        dropout=0.1,
    ):
        ...
```

Inputs:

```text
history frame latents
past action tokens
optional pose tokens during training only
```

Outputs:

```text
memory_tokens: [B, M, D]
```

Where:

```text
M = fixed memory budget, e.g. 4, 8, 16, 32
D = memory dimension
```

### 5.2 Bottleneck

Implement at least:

```text
fixed number of memory tokens
token dropout
optional L2 regularization
optional variance regularization
```

Do not begin with complicated VQ. Add VQ only after the basic method works.

### 5.3 Future-test decoder

Implement a decoder that receives:

```text
memory_tokens
future action suffix tokens
test_type embedding
```

and predicts different heads:

```text
future frame embedding
loop return binary label
matched old frame distribution
landmark visible binary label
object exists binary label
```

Suggested heads:

```python
future_embedding_head: MLP -> D_embed
loop_return_head: MLP -> scalar logit
match_history_head: cross-attention over stored history indices or classification over candidate frames
landmark_visible_head: MLP -> scalar logit
object_exists_head: MLP -> scalar logit
```

### 5.4 How PTM affects generation

The future-test heads are not enough. The memory must also condition the world-model rollout.

Add memory conditioning into WorldMem/Oasis-family generation by one of these methods:

Preferred:

```text
cross-attention from world-model tokens to PTM memory tokens
```

Alternative:

```text
prepend memory tokens to conditioning sequence
```

Fallback:

```text
adapter that maps memory tokens into the existing memory-bank interface
```

Acceptance criterion:

```text
generated rollouts must change when memory tokens are ablated
```

Add an ablation script:

```text
normal memory vs zeroed memory vs shuffled memory
```

## 6. Training losses

### 6.1 Base world-model loss

Keep the original WorldMem/Oasis-family generation loss.

Do not remove generation training.

### 6.2 Future-test losses

Use:

```text
L_future_tests =
  w_embed * L_future_embedding
+ w_loop * BCE(loop_return)
+ w_match * CE(matched_history_t)
+ w_landmark * BCE(landmark_visible)
+ w_object * BCE(object_exists)
```

Suggested initial weights:

```yaml
w_embed: 1.0
w_loop: 0.5
w_match: 0.5
w_landmark: 0.5
w_object: 0.5
lambda_test: 0.25
beta_bottleneck: 0.001
```

Tune only after smoke tests.

### 6.3 Embedding target

For target future frame embedding, use one of:

```text
DINOv2 embedding
CLIP/MineCLIP embedding
Oasis VAE latent
WorldMem internal latent
```

Start with the internal VAE/world-model latent to avoid adding too many dependencies. Add DINO/CLIP metrics for evaluation later.

### 6.4 Leakage prevention

Critical rules:

```text
future-test decoder cannot see future frames
future-test decoder cannot see full raw history
future-test decoder reads only memory_tokens plus future actions plus test_type
pose can be used to generate labels
pose should not be given to the model at inference unless explicitly testing a pose-conditioned variant
```

Add baselines:

```text
pose-only predictor
visual-memory-only predictor
action-only predictor
```

This prevents a reviewer from saying the task is solved by coordinates.

```text
  similarity(generated_return_frame, generated_initial_view_of_same_place)

  predicted/observed return matches correct old frame
```

Report:

```text

```

### 9.3 Landmark persistence metrics

For landmark episodes:

```text
landmark visible at return
landmark identity preserved
landmark approximate position preserved
```

Use programmatic labels if possible. If using simple colored block landmarks, implement a color/block-pattern detector.

Report:

```text
landmark hit rate ↑
landmark identity accuracy ↑
false disappearance rate ↓
```

### 9.4 Placed-object persistence metrics

For placed-object episodes:

```text
object exists in generated return view
object type is correct
object not hallucinated before placement
```

Report:

```text
object persistence accuracy ↑
false disappearance ↓
false hallucination ↓
```

### 9.5 Memory budget curve

Run with:

```text
num_memory_tokens = 4, 8, 16, 32
```

Report:

```text
same memory budget, PTM improves more than compressed/retrieval memory
```

This is essential to support the claim:

```text

```

## 13. Reviewer-risk mitigation

### Risk: "This is just auxiliary loss."

Mitigation:

```text
Compare PTM vs future-frame-only auxiliary loss.
Show PTM improves loop/object/landmark metrics more.
Show ablations removing test families reduce corresponding capability.
```

### Risk: "The dataset is scripted."

Mitigation:

```text
Randomize seeds, biomes, path shapes, camera jitter, landmark types, object types.
Use train/test splits with held-out biomes and held-out landmark layouts.
Report harder split separately.
```

### Risk: "The model cheats with pose."

Mitigation:

```text
Pose-only baseline.
No-pose inference variant.
Visual-only memory variant.
Do not feed absolute pose into generation unless explicitly comparing pose-conditioned models.
```

### Risk: "This is just WorldMem."

Mitigation:

```text
Use WorldMem-style memory bank as baseline.
Keep memory budget matched.
Change the training definition: future-test sufficiency instead of only memory retrieval/reconstruction.
Report memory budget curve.
```

## 14. Recommended first implementation order

Do not start by modifying the full model.

Follow this order:

```text
1. Generate 10 MineDojo episodes and view frames.
2. Generate 100 labeled episodes.
3. Implement dataset loader.
4. Train PTM heads only on frozen frame embeddings.
5. Verify PTM heads learn.
6. Connect PTM memory to generation.
7. Train small adapter.
8. Compare to compressed-memory baseline.
9. Add WorldMem-style baseline.
10. Scale data.
```

This order avoids wasting time on heavy model integration before the data and labels are proven.

## 16. References and source facts

These are the source facts this implementation plan relies on.

1. **Open-Oasis / Oasis 500M**
   - Repo: https://github.com/etched-ai/open-oasis
   - Hugging Face: https://huggingface.co/Etched/oasis-500m
   - Fact used: Oasis 500M weights and inference code are released for action-conditioned frame generation.

2. **WorldMem**
   - Repo: https://github.com/xizaoqu/WorldMem
   - Project: https://xizaoqu.github.io/worldmem/
   - Fact used: WorldMem is based on Open-Oasis architecture, uses pretrained VAE and DiT weights from Open-Oasis, includes training/evaluation scripts, and uses MineDojo to collect Minecraft data.

3. **MineDojo**
   - Docs: https://docs.minedojo.org/
   - Observation space: https://docs.minedojo.org/sections/core_api/obs_space.html
   - Fact used: MineDojo can provide egocentric RGB frames, inventory, inventory change, voxels, life statistics, location statistics, and action spaces suitable for controlled trajectory generation.

4. **PLAICraft**
   - Paper: https://arxiv.org/abs/2505.12707
   - Public data blog: https://blog.plaicraft.ai/2025/05/18/public-datasets-now-available/
   - Fact used: PLAICraft contains time-aligned Minecraft video, audio, mouse, and keyboard actions, but it is not the recommended first-stage PTM training data because PTM needs clean pose/event/future-test labels.

5. **WorldPack**
   - Paper: https://arxiv.org/abs/2512.02473
   - Fact used: nearby memory work evaluates spatial consistency and long-term memory in Minecraft/LoopNav-style settings.
